# Prhythm バンドメンバー検証ヒアリング分析フレーム

対象スキル: `hearing-analysis`  
前提: 実ヒアリング後に埋める分析テンプレート。現時点では persona 文書をもとにした仮説を入れているため、**[Assumption]** が中心。

## 1. 顧客の目的

**[Assumption]** バンドメンバーは、案件初期のヒアリング・整理・提案準備を、個人の頭の中で終わらせず、後から参加するメンバーやリーダーも使える材料にしたい。

## 2. 現状

- 現在の業務フロー:
  - **[Assumption]** ヒアリング前に Slack / Notion / 過去資料で準備する
  - **[Assumption]** 会議後に議事録や個人メモを残す
  - **[Assumption]** 必要に応じて提案方針や対象ユーザーを後から整理する
- 関係者:
  - バンドメンバー
  - バンドリーダー
  - バンドマネージャー
  - 後から参加するメンバー
- 使用ツール:
  - **[Assumption]** Slack、Notion、議事録、汎用AI、Miro
- 時間や頻度:
  - **[Unknown]** 実案件での頻度と所要時間は未確認

## 3. 表面課題

- **[Assumption]** ヒアリング前に何を聞くべきかが人によってずれる
- **[Assumption]** 会議後の議事録が、次の判断に使える論点へ変換されない
- **[Assumption]** 後から入るメンバーが文脈を追いにくい
- **[Assumption]** スキル連鎖が重く見えると、汎用AIや個人メモに戻る

## 4. ボトルネック仮説

| 仮説 | 詰まり方 | まだ未確認の前提 |
|---|---|---|
| 情報が「記録」で止まり「判断材料」にならない | 情報不足 / 構造化不足 | 議事録から何を再利用しているか |
| チーム資産化より個人速度が優先される場面が多い | 時間制約 / 現場受容 | どの場面で短さが最優先になるか |
| 読み手が案件ごとに変わり、成果物の粒度が揺れる | 責任分界 / 読み手不明 | 誰が成果物を読むべきか |
| 全スキル連続利用の期待が重すぎる | 現場受容 | 単発利用からどこまで広がるか |

## 5. 根拠

- 発言ベースの根拠:
  - **[Fact]** バンド未組成案件では個人が先行してヒアリング・提案に入ることがある
  - **[Fact]** バンド組成や運用ルールは未確定
- 観測ベースの根拠:
  - **[Unknown]** 実利用観察は未実施
- こちらの推測:
  - **[Assumption]** 「チーム資産化」は価値だが、初回利用では重さが勝つ可能性がある

## 6. 解決方向

- 業務整理:
  - ヒアリング前後の最低限の成果物を定義する
- 情報集約:
  - 議事録、論点、未確定事項、次回確認事項を分ける
- 標準化:
  - すべてを標準化せず、2〜3 スキル連携の軽い導線を作る
- AI 活用:
  - 自動化前提ではなく、判断材料の変換支援として使う

## 7. 初期スコープ案

最初の検証では、次の 2 パターンだけを見る。

| パターン | 試すスキル | 見ること |
|---|---|---|
| ヒアリング前後 | `hearing-prep` → `hearing-analysis` | 会議前後の整理が次回判断に使えるか |
| 提案整理 | `hearing-analysis` → `defining-personas-and-segments` → `product-vision-and-concept` | 対象と提案の芯が絞れるか |

## 8. 次回確認事項

- どの成果物なら 10 分以内に読めるか
- 誰が成果物の主な読み手か
- 単発利用から 2 つ目のスキルへ進む条件は何か
- 個人先行案件でも使える軽い入口は何か
- 管理者向け標準化ニーズは現場ニーズより強いか

## 9. 擦り合わせ質問集

| カテゴリ | 質問 | どのズレを確認するためか | 回答で何を判断するか |
|---|---|---|---|
| 目的の擦り合わせ | Prhythm に期待するのは「自分の準備を速くすること」と「チームに残すこと」のどちらに近いですか | Primary 仮説の中心 | 個人速度かチーム資産化か |
| 現状フローの擦り合わせ | 会議後のメモは、次の提案判断で実際に参照されていますか | 記録と判断材料のズレ | hearing-analysis の価値 |
| ボトルネック仮説の擦り合わせ | 後から入った人が困るのは、情報量不足ですか、判断理由不足ですか | 情報不足か構造化不足か | 成果物の重点 |
| 制約 / 運用の擦り合わせ | 何分を超えると「重い」と感じますか | 現場受容 | 出力長と導線設計 |
| 解決方向の擦り合わせ | 2つ目のスキルに進むなら、どの成果物が入力として自然ですか | スキル連鎖の実在性 | 導線の優先順位 |

## 10. 聞かない / 約束しない方がよいこと

- Prhythm の全スキルを必ず順番に使う前提では聞かない
- 「標準化できます」と約束しない
- バンド運用ルール自体を Prhythm が解くとは言い切らない
- 実装接続系スキルまで価値が出るとはまだ約束しない
