# validation-log

> プロト検証の観察・計測結果を構造化された台帳 `docs/validation-log.md` に記録するエージェントスキル。各エントリは **Scale + Period + Outcome** の 3 要素を備え、A ID に紐付く。`uncertainty-map` が ⬜ → 🟡 → ✅ の昇格判定に使う観察証拠の **供給元** となる。

ユーザーテスト・A/B・スパイク・ベンチマークといった検証アクティビティの結果を、価値検証（#1-9）／技術検証（#10-14）の両対応で記録します。「動いた」「だいたい」のような曖昧な記述を排し、機械可読な計測値（完了率・CVR・p95・有効率・CLS・成功数）で残すことで、`uncertainty-map` の `✅ 検証済` 昇格が自動的に根拠を拾えるようにします。

- **入力:** 検証アクティビティの結果（ユーザーから口頭/メモ）/ `docs/uncertainty-map.md`（A ID 突き合わせ、推奨）/ 既存 `docs/validation-log.md`（追記時）
- **出力:** `docs/validation-log.md`（時系列の検証エントリ台帳 + 昇格サマリ）
- **スコープ外:** 検証アクティビティの実施・実装そのもの（別タスク）/ 仮説の抽出・マトリクス化（→ `uncertainty-map`）/ マップの昇格確定（→ `uncertainty-map` の次回実行）

---

## 概要

検証の結果は、記録の仕方が雑だと「実装した」と「ユーザーで確かめた」の境界が溶け、後から `✅ 検証済` の根拠を再構成できなくなります。本スキルは各検証結果を **V ID** を持つ追記専用エントリとして記録し、必須 3 要素（Scale + Period + Outcome）・検証手段（#1-14）・失格条件・合否判定・学びを構造化します。出力は Markdown のため GitHub・VS Code でそのまま閲覧・差分レビューでき、`uncertainty-map` の脚注（`docs/validation-log.md#V-01`）からも参照できます。

> **責務分離**: 本スキルは証拠を **記録するだけ** です。仮説ステータスの昇格（🟡 → ✅）は `uncertainty-map` が次回実行時に、このログを根拠として判定します。本スキルは `docs/uncertainty-map.md` を書き換えません。

## 利用メリット

- **「動かした」と「検証した」が混ざらない** — Scale + Period + Outcome の 3 要素が揃わないエントリは `要追加計測` として記録され、✅ 級の根拠にならないので、検証ステータスが過大評価されない
- **価値検証も技術検証も同じ台帳に残せる** — ユーザーテスト（#1-9）とスパイク・ベンチマーク（#10-14）を value/technical フラグで区別しつつ一元管理できる
- **検証の軌跡が消えない** — 追記専用で過去エントリを書き換えないので、「いつ・何を・どれだけの規模で確かめたか」の履歴が保全される
- **昇格が自動で根拠を拾える** — V ID と A ID を保つことで、`uncertainty-map` の昇格判定が機械的に Scale/Period/Outcome を参照できる
- **対外レポートの脚注が壊れない** — Mode B レポートの脚注がこのログの V ID を指すので、検証済成果の出典が常に追跡可能

## 利用シーン

- **ユーザーテストやインタビューを終えて、結果を残したいとき** — 完了率・平均時間・課題言及率などを構造化して記録する
- **エンジニアリングスパイクや PoC を試した直後** — 何ケース中何件成功したか、制約は何かを機械可読に残す
- **性能ベンチマーク・AI 出力の品質計測をしたとき** — p95・有効率・CLS などを失格条件と並べて判定する
- **`uncertainty-map` を再実行して昇格させる前** — 昇格の根拠となる観察ログを先に整える
- **「動いた」で済ませそうになったとき** — 機械可読な計測値を要求し、無ければ `要追加計測` に倒す

依頼の例:

```
A-D01-01-01 のユーザーテスト結果（5 人 / 完了率 100% / 平均 47 秒）を検証ログに記録して
DOM→pptx スパイクの結果（3 ケース中 3 件成功）を A-D02-06-T1 の証拠として残して
LP スモークテスト（訪問 120 / CVR 4.2%）を追記して
```

## 使い方

Claude Code / Cursor 上で次のように依頼すると起動します。

```
/validation-log A-D01-01-01 のユーザーテスト結果を記録して
/validation-log スパイクの結果を検証ログに残して
```

スラッシュ無しでも、検証ログ・観察ログ・ユーザーテスト結果を記録・スパイク結果・ベンチマーク結果・validation log などのキーワードを含む依頼で自動起動します。

`docs/uncertainty-map.md` がある状態だと、エントリを既存の A ID に正確に紐付けられます。検証アクティビティの結果（人数・期間・計測値）を伝えると、3 要素を構造化して記録します。

## 出力例（抜粋）

**`docs/validation-log.md` のエントリ**

```markdown
### V-03 — メール登録動線
- 日付: 2026-06-12
- 対象 A ID: A-D01-01-01
- 種別: value
- 仮説: ターゲットは email 登録に抵抗がなく 1 分以内に完了できる
- 検証手段: #8 5 ユーザーテスト
- Scale: n=5 users
- Outcome: 完了率 100% / 平均 47 秒
- 失格条件: 完了率 < 80% または 平均 > 120 秒
- 判定: 合格
- 学び: PW マネージャ利用者はさらに速い（平均 31 秒）
- 次アクション: A-D01-01-01 を ✅ 昇格候補へ
- 生データ: docs/usability/2026-06-12-signup.md
```

**昇格サマリ**

```markdown
| A ID | 現ステータス（map） | このログの根拠 | 昇格可否 |
|------|--------------------|---------------|---------|
| A-D01-01-01 | 🟡 | V-03 (n=5 / 完了率 100%) | ✅ 昇格可 |
| A-D03-02-T1 | 🟡 | V-05 (要追加計測) | 据え置き |
```

詳細テンプレートは [`references/log-template.md`](references/log-template.md) と [`references/entry-schema.md`](references/entry-schema.md) を参照。

## 前提条件

- Claude Code または Cursor（プラグイン `prhythm` の一部として配布）
- 推奨入力 `docs/uncertainty-map.md`（[uncertainty-map](../uncertainty-map/) で生成）— A ID 突き合わせのため
- 検証アクティビティ（ユーザーテスト / スパイク / ベンチ等）の結果データ — 人間が実施した成果

## 注意事項

- **3 要素が揃わなければ ✅ 級にしない** — Scale / Period / Outcome のいずれか欠落は `要追加計測` として記録し、`uncertainty-map` は 🟡 据え置き
- **Outcome は機械可読な数値** — 「動いた」「だいたい」「数人が好評」は不可。完了率 / CVR / p95 / 有効率 / CLS / 成功数で書く
- **追記専用** — 過去エントリは書き換えない。訂正は新エントリ + 相互リンク
- **マップは書き換えない** — 昇格は提案（surface）まで。確定は `uncertainty-map` の次回実行
- **数値の捏造禁止** — 人数・期間・計測値は確認できないものは `—`
- 出力は Markdown のみで、外部の検証ツール連携や自動計測は行いません
