## Purpose

このスキルは、FDEが顧客との対話を起点に、価値仮説とリスク仮説を更新し続けるためのディスカバリーループである。

主目的は、最終提案書や包括的な技術検討資料を作ることではない。

主目的は、顧客・MD・コンサルから得た断片情報をもとに、次回ヒアリングで何を聞くべきかを絞ることである。

一言で言うと、顧客理解を、開発が判断できる問いと仮説に変換し続けるスキル。

## Positioning

このスキルは、コンサル型ヒアリングの代替ではない。

コンサル型ヒアリングが得意なこと:

- 顧客の痛みを引き出す
- 顧客の進歩や購買動機を理解する
- 業務/経営課題として整理する
- 提案ストーリーを作る

このスキルが担うこと:

- 顧客の言葉を、開発観点の確認論点に変換する
- 提示された解決策が本当に妥当かを疑う
- ボトルネックを特定する楔となる問いを作る
- 技術的に確認しないと危ない前提を出す
- コンサル/MDに共有すべき注意点を短く出す

このスキルは、コンサル型ヒアリングで得た顧客理解を補完し、プロダクト開発・技術リスク・PoC設計に入る前のヒアリング論点を整理するための入口である。

## Scope

このスキルのコアスコープは狭く保つ。

やること:

1. 入力情報を要約する
2. ボトルネック仮説を出す
3. 開発観点で聞くべき問いを出す
4. 問いの根拠を出す
5. 顧客向けにそのまま聞ける形へ変換する
6. コンサル/MDに共有すべき注意点を短く出す
7. ヒアリング後に回すべき別スキルを示す

やりすぎないこと:

- 詳細な技術設計
- 本格的なPoC設計
- 試作内容の決定
- 提案内容の決定
- 顧客検証の実施や評価
- 業務定着計画
- 提案書作成
- 詳細なMD Decision
- 詳細な本番化リスク整理
- コンサル向けの長いハンドオフ資料

## Minimal Workflow

このスキルは単発の質問生成ではなく、ヒアリング内で仮説と問いを更新するループとして回す。

1. 情報を受け取る
    - 顧客/MD/コンサルから、業務・痛み・期待・不安・制約を受け取る
2. 構造化する
    - 誰が、どこで、なぜ詰まっているのかを整理する
3. 仮説化する
    - 価値仮説とリスク仮説に分ける
4. 質問を作る
    - 次回ヒアリングで、仮説の真偽を判定できる問いに絞る
5. ヒアリング結果で更新する
    - 追加で得た回答をもとに、次に聞くべき問い・確認すべき前提・避けるべき約束を更新する

このスキルの責務は、ヒアリング設計とヒアリング結果に基づく問いの更新までである。

試作・提案・顧客検証は、このスキルの責務に含めない。

## Input

入力は断片でよい。

例:

- 顧客の発言
- MD/コンサルの仮説
- 提示されている解決策
- 既知の業務課題
- 顧客資料
- 営業メモ
- ヒアリング前の仮説
- ヒアリング後の文字起こし

入力が少なくても、分かる範囲で問いを出す。

ただし、情報不足なのに解決策を断定しない。

## Output Format

原則として、以下だけを出す。

### 1. 入力の要約

- 顧客が言っていること:
- 既に提示されている解決策:
- 開発観点で気になること:

### 2. 価値仮説

- 本当に詰まっていそうな箇所:
- それが業務インパクトにつながりそうな理由:
- まだ分からないこと:

### 3. リスク仮説

- 期待値ズレが起きそうな点:
- 実現不能な約束になりそうな点:
- 現場反発・運用不全が起きそうな点:
- 先に確認しないと危ない前提:

### 4. 次に聞くべき問い

最大5問まで。

1. 質問:
    - なぜ聞く:
    - 回答で判断できること:
    - 対応する仮説: 価値仮説 / リスク仮説
2. 質問:
    - なぜ聞く:
    - 回答で判断できること:
    - 対応する仮説: 価値仮説 / リスク仮説
3. 質問:
    - なぜ聞く:
    - 回答で判断できること:
    - 対応する仮説: 価値仮説 / リスク仮説

### 5. 顧客向けスクリプト

顧客にそのまま聞ける形にする。

- 最大7問
- 1問1文
- 技術用語を使いすぎない
- 質問の意図は書かない
- 顧客が答えやすい順番に並べる

### 6. 聞かない / 約束しない方がよいこと

- 今は聞かない方がよいこと:
- まだ解決策として約束しない方がよいこと:
- 提案っぽく言い切らない方がよいこと:

### 7. ヒアリング後に更新された理解

ヒアリング後の入力がある場合のみ出す。

- 更新された価値仮説:
- 更新されたリスク仮説:
- 次回ヒアリングで優先する問い:
- 後続スキルに渡すべき論点:

### 8. ヒアリング後に回すべき別スキル

必要なものだけ示す。

- fde-technical-critique:
- fde-mtp-planner:
- fde-workflow-adoption-checker:
- fde-consultant-handoff:

## Question Design Principles

質問は、単なるヒアリング項目ではなく、ボトルネックを特定する楔にする。

悪い質問:

- 何に困っていますか？
- どんなツールが欲しいですか？
- AIで何をしたいですか？

良い質問:

- 最後にその問題で作業が止まった具体例は何ですか？
- その作業の中で、人間の判断が必要だったのはどこですか？
- それは情報不足で止まったのか、判断待ちで止まったのか、作業量で詰まったのか？
- その作業を他の職種に渡せない理由は何ですか？
- AIが初稿を出しても使われないとしたら、何が原因になりそうですか？

## FDE Lens

質問を作るときは、以下を見る。

- 誰が使うのか
- どの業務フローに入るのか
- 使わない理由は何か
- 精度よりUXが悪いのか
- データが足りないのか
- 権限・運用・責任分界で止まっているのか
- AIで作れるものではなく、AIで変えられる業務はどこか

## Handoff to Other Skills

このスキルは入口に限定する。

必要に応じて、後続スキルへ渡す。

### fde-technical-critique

目的:

提示された解決策が技術的に妥当かを批判的に見る。

使うタイミング:

ヒアリング後、顧客の課題と解決策候補がある程度見えたとき。

出力例:

- 妥当な点
- 怪しい点
- 技術的懸念
- 代替案

### fde-mtp-planner

目的:

最小検証プロダクトに落とす。

使うタイミング:

何を試すべきかが見えてきたとき。

出力例:

- 入力
- 出力
- 使う人
- 使う場面
- 成功条件
- 失敗した場合に分かること

### fde-workflow-adoption-checker

目的:

作ったものが業務に入るかを見る。

使うタイミング:

PoC案またはプロダクト案が出た後。

出力例:

- 実利用者
- 利用タイミング
- 既存業務のどこに入るか
- 使われない理由
- 定着条件

### fde-consultant-handoff

目的:

FDEの見立てをMD/コンサル/開発に渡せる言葉に変換する。

使うタイミング:

ヒアリング後、社内共有や提案前整理が必要なとき。

出力例:

- 現在の仮説への見立て
- 技術観点での懸念
- 先に確認すべきこと
- 提案上の言い換え
- 次回MTGで拾ってほしい論点

## Anti-Patterns

避けること。

- 顧客要望をそのまま要件化する
- 「AIを使いたい」をそのままAI開発提案にする
- 質問を大量に出す
- コンサルがやるべき課題整理まで抱え込む
- 初回ヒアリング前に詳細なPoC設計まで出す
- 本番化リスクを過剰に列挙して出力を重くする
- 顧客向け質問に技術用語を入れすぎる
- MD/コンサル向けに長い資料を毎回出す

## Final Principle

このスキルは、FDEのディスカバリーループを支援する。

- 顧客には軽く聞く
- FDE内部では深く疑う
- 価値仮説とリスク仮説に分けて考える
- 開発観点で刺さる問いに絞る
- ヒアリング結果をもとに次の問いを更新する
- 後続の詳細検討は別スキルに渡す

最終目的は、FDEが「作れそうなもの」ではなく「業務を変えられるボトルネック」を見つけるための問いを設計し、同時に期待値ズレ・実現不能な約束・現場反発による炎上を避けるための前提確認をできるようにすること。

## Output Rule: 実装時に詰まりそうな点

`コンサル/MDへの注意点` の中にある「実装時に詰まりそうな点」を出す場合は、抽象的なリスク名だけで終わらせない。

必ず以下の粒度で端的に書く。

1. 何を作る想定か
2. なぜそこで詰まりそうか
3. 代替案は何か

### 出力フォーマット

```markdown
### 実装時に詰まりそうな点

#### 1. {実装シナリオ名}

何を作る想定か:
- 

なぜ詰まりそうか:
- 

代替案:
- 
```

### 書き方のルール

- 技術スタックやツール名が想定できる場合は具体名を書く
- ただし、技術比較表にはしない
- 「なぜ詰まるか」の根拠を書く
- 「だから何を切るべきか」まで書く
- MD/コンサルが提案や次回MTGで使える言葉にする
- 長くなりそうな場合はトグルに閉じる

### 良い例

```markdown
#### PowerPoint完成品まで自動生成する案

何を作る想定か:
- LLMで構成案と本文を生成し、PptxGenJSなどで .pptx を自動生成する。
- 顧客提出に近いPowerPoint資料を出す。

なぜ詰まりそうか:
- .pptx生成自体は可能だが、顧客向け資料として使えるレイアウト品質を安定させるのが難しい。
- 文字量、余白、図表配置、ページ分割、デザイン崩れの調整が必要になる。
- 「完成資料」を約束すると、技術正確性だけでなく見た目の品質責任まで開発側に乗る。

代替案:
- 初手では .pptx 完成品を作らない。
- 出力は「構成案 + スライド別本文ドラフト + 図表案」までにする。
- PowerPoint化は人間が既存テンプレートに貼るか、後続フェーズで検証する。
```

### 悪い例

```markdown
- PowerPoint生成は難しい
- RAGは大変
- 権限管理が必要
```

これはリスク名の羅列であり、FDEのハンドオフとしては不十分。

### 意図

実装時に詰まりそうな点は、開発側が「後で困りそう」と感じる論点を、MD/コンサルが理解できる形に翻訳するためのもの。

そのため、単なる技術メモではなく、以下まで含める。

- 何を作ろうとしているのか
- どの実装で詰まりそうなのか
- なぜ詰まりそうなのか
- どこまでを初回PoCから外すべきか
- 代替案として何を提案すべきか

これにより、提案時点でスコープ爆発や過剰約束を防ぐ。
