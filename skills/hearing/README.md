# hearing

> FDEが顧客との対話を起点に、**価値仮説とリスク仮説を更新し続ける**ディスカバリーループスキル。

顧客・MD・コンサルから得た断片情報をもとに、次回ヒアリングで何を聞くべきかを絞る。顧客理解を、開発が判断できる問いと仮説に変換し続けることが目的。

---

## 何を生成するか

| 出力 | 内容 |
|---|---|
| **入力の要約** | 顧客の発言・提示済み解決策・開発観点で気になること |
| **価値仮説** | 本当に詰まっていそうな箇所・業務インパクト・未知の情報 |
| **リスク仮説** | 期待値ズレ・実現不能な約束・現場反発・危ない前提 |
| **次に聞くべき問い** | 最大5問。各問に「なぜ聞く」「回答で判断できること」「対応する仮説」を付与 |
| **顧客向けスクリプト** | そのまま顧客に聞ける形の質問（最大7問） |
| **聞かない / 約束しない方がよいこと** | 今は避けるべき質問・約束・言い切り |
| **ヒアリング後に更新された理解** | ヒアリング後の入力がある場合のみ出力 |

## 使い方

Claude Code 上で以下のように依頼すると起動します。

```
/hearing 顧客が品質レポートの自動生成をしたいと言っている。月3日かかっている
ヒアリング準備して。営業メモはこれ: ...
次回ヒアリングの質問を考えて
```

ヒアリング後に追加情報を渡すと、仮説と問いが更新されます。

```
ヒアリング結果: データ収集に2日、考察に1日かかっていることが分かった
```

## 入力

断片情報でよい。以下を単独・複数組み合わせて入力できます。

| 入力種別 | 例 | 主に埋まる情報 |
|---|---|---|
| 顧客の発言 | 「レポートに3日かかる」「AIで自動化したい」 | 痛み・期待 |
| MD/コンサルの仮説 | 「RAGでいけるのでは」「業務フロー可視化から」 | 提示済み解決策 |
| 既知の業務課題 | 業務フロー図、課題一覧 | ボトルネック候補 |
| 顧客資料 | 現行レポート、Excel データ | データ構造・フォーマット |
| 営業メモ | 商談メモ、議事録 | 顧客の温度感・制約 |
| ヒアリング後の文字起こし | MTG録音のテキスト化 | 仮説更新の材料 |

## スコープ

**やること:**

- 入力情報の要約
- ボトルネック仮説の生成
- 開発観点で聞くべき問いの設計
- 顧客向け質問への変換
- コンサル/MDへの注意点の短い共有

**やりすぎないこと:**

- 詳細な技術設計・PoC設計
- 提案内容の決定・提案書作成
- 顧客検証の実施や評価
- コンサル向けの長いハンドオフ資料

## 動作環境・前提

- Claude Code（プラグイン `prhythm` の一部として配布）
- Web 検索は不要。入力された断片情報のみで動作する
