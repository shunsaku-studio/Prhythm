# hearing-analysis

> ヒアリング後の情報を整理し、ボトルネック仮説と次回の擦り合わせ質問を出すスキル。

## 概要

会議メモや文字起こし、事前仮説をもとに、会議後にはじめてボトルネック仮説を立てる。顧客の発言とこちらの見立てを分け、次回ヒアリングで何を擦り合わせるべきかを整理する。

## 利用メリット

- **会議後の整理が構造化される** - 目的、現状、表面課題、ボトルネック仮説を分けて残せる
- **仮説と事実が混ざりにくい** - 発言ベースの根拠、観測、推測を分離して書ける
- **次回ヒアリングにつながる** - 擦り合わせ質問集まで一緒に出る
- **旧 hearing 資産を継承できる** - 開発観点での確認論点やリスク仮説を引き続き使える

## 利用シーン

- **ヒアリング直後に議事録を整理したいとき** - 顧客の発言と見立てを分けて残す
- **こちらの仮説が合っているか次回確かめたいとき** - 擦り合わせ質問集を作る
- **解決方向を広めに整理したいとき** - いきなり実装案に飛ばず、方向性レベルに留める
- **複数回のヒアリングで仮説更新したいとき** - 毎回の差分を次回確認事項に落とせる

## 使い方

```text
/hearing-analysis
ヒアリング前の仮説: レポート文章生成が重いのではないか
ヒアリング結果: データ収集に2日、考察に1日。文章化自体は30分程度
次回の擦り合わせ質問もほしい
```

## 構成

```text
hearing-analysis/

```

## 前提条件

- 会議メモ、文字起こし、またはヒアリング前準備の出力があること
- 事実と仮説を分けて扱う前提で使うこと
- 会議後の整理であり、会議前の進行設計ではないこと

## 注意事項

- 実装案、技術設計、製品提案をここで確定しない
- AI / 自動化を前提にした結論へ飛ばない
- 擦り合わせ質問は誘導ではなく、仮説が外れていたら外れていたと分かる形にする
