# 出力例 — Stage 2（深掘り）

題材: 介護施設の引き継ぎノート（B2B 社内ツール）。
**前提**: Stage 1（`output-example-stage1.md`）の出力を受けて、人間が「Primary は Segment 1（既に書く習慣がある現場リーダー）」と指定したシナリオ。

---

## 1. Persona Deep Dive

### ペルソナ 1: 紙で書いている現場リーダー（属人化を解消したい）

#### 対応するセグメント
Segment 1（既に書く習慣がある現場リーダー）

#### サマリー
3〜5人の介護スタッフを束ねる現場リーダー。日々の引き継ぎを紙ノートと LINE グループで運用しており、書くこと自体は5年以上やってきて速くこなせる。一方で「あとから検索できない」「夜勤者がノートを見落とす」ことに不満を持っており、属人化と情報の流れに穴がある状態に気づいている。エクセルや新しいツールを試したことはあるが続かなかった。

#### 典型的な利用シーン
- **18:30** — 夕勤の終わり際、夜勤者の出勤直前。当日昼間の利用者の様子（特に体調変化やヒヤリハット）を急いでまとめている。
- **翌朝 7:30** — 夜勤者と日勤者の交代時。「あれ書いたっけ」と前日のノートを探すが、ページが分かれていて見つからない。
- **週1回の振り返り会** — 過去のヒヤリハットを話し合うが、検索できないのでうろ覚えで話すしかない。

#### JTBD 階層

| 階層 | 説明 | 内容 |
|---|---|---|
| Main job | プロダクトに「雇う」中心的な仕事 | **[Fact]** 夜勤者に必要な情報を確実に伝える（3人中3人言及） |
| Related job | Main の周辺で片付けたい仕事 | **[Assumption]** 過去のヒヤリハットを見返したい（A のみ言及、B は「溜まるが見ない」と逆） |
| Emotional job | 感情面の動機 | **[Assumption]** 「自分が休んでも回る」という安心感を得たい |
| Social job | 他者からどう見られたいか | **[Assumption]** 後輩・新人に「ちゃんと申し送りしているリーダー」と見られたい |

#### 人物の具体像（行動・選好の手がかり）

| 項目 | 内容 | 設計判断への効き方 |
|---|---|---|
| 名前（仮） | 山田 紀子 | 議論で具体名のほうが想起しやすい。**[Risk]** 「介護=女性」ステレオタイプの固定化 |
| 年齢・属性 | **[Fact]** 42歳、女性、介護施設7年（A・B の年代帯から） | 既存習慣の強さの初期値、紙文化への愛着の推定材料 |
| 性格特性 | **[Assumption]** 慎重、ミスを恐れる、決める前にベテランに相談したい | エラーダイアログのトーンを「責める」より「支える」寄りに |
| 決定スタイル | **[Fact]** 自分一人では決めない、施設長承認が必要（U5） | 「導入決定」フローに施設長向け情報設計が要る |
| ストレス時の反応 | **[Assumption]** 急ぐと紙に戻る、新規学習を放棄する | 学習コスト最小化、紙の自由度を一定担保 |
| 普段の口調 | **[Assumption]** 丁寧語、感情を抑えめに表現 | 通知・確認メッセージのトーンも丁寧寄り |
| 好きなもの / 慣れているもの | **[Fact]** 紙ノート、LINE（緊急連絡） | LINE 風UI / 紙ノート的な自由入力 が参照点 |
| 苦手なもの / 避けるもの | **[Fact]** エクセル関数（過去に挫折）、横文字メニュー | 用語は和語寄り、機能名にカタカナを使わない |
| 平日のリズム | **[Assumption]** 朝6:30起床、家事→出勤(8:30)→17時上がり、夜は家族と | プッシュ通知は勤務時間内のみ。夜間の業務外通知は避ける |

**[Risk]** ここの項目のうち、性格・口調・好きなものは A 1人の発話パターンから一般化している。Stage 2 の Validation Plan で B にも確かめる。

#### インタビュー引用 + 推測の境界
- **[Fact]** 「夜勤の人がノート見落とすことがあって、ヒヤッとしたことがある」（A 直接発言）
- **[Fact]** 「書くのは慣れてるから速い。エクセルは続かなかった」（A 直接発言）
- **[Assumption]** 「2分以上かかると諦めそう」 — A は「すぐ書ける」と言ったが具体的な秒数は言っていない。書き手にとっての時間しきい値は推測
- **[Assumption]** 「夜勤者の既読が分かるとモチベーションが上がる」 — A は「見落とされて困る」と言ったが、既読確認が解決策として響くかは未検証

#### 現在の代替手段の詳細
- **ツール**: 紙ノート（A4 ルーズリーフ綴じ）+ LINE グループ（緊急時のみ）
- **不満点**: 検索できない / 夜勤者が読んだかどうか分からない / 古い情報がページに紛れる
- **満足している点**: すぐに書ける / 考えなくていい / フォーマットの自由度が高い / 停電や端末故障に強い
- **なぜ完全に置き換えられないか**: **[Risk]** 紙の自由度を犠牲にすると、書き慣れた速さを失う可能性

#### バリアと離脱条件

- **心理的**:
  - **[Assumption]** 「新しいツールは続かない」という諦め（過去のエクセル運用が消えた経験）
  - **[Unknown]** 「自分の書き方を否定された」と感じると拒否する可能性

- **実務的**:
  - **[Assumption]** 入力に2分以上かかると諦める（具体的な秒数は要検証）
  - **[Fact]** PCもタブレットも共用端末。ログイン管理ができないと回らない

- **政治的**:
  - **[Fact]** 施設長が承認しないと導入できない（U5 ヒアリング）
  - **[Assumption]** 「家族からツールが見えるとクレームのリスク」と懸念する可能性

#### 成功の定義
- 夜勤者から「これ見て助かった」と言われる
- 自分が休んだ日でも、後日「ちゃんと回ってた」と確認できる
- 過去のヒヤリハットを月1回の振り返りでサッと引ける

#### 失敗の定義
- 3回入力したのに夜勤者の反応がない
- 同僚に勧めたが「面倒だから紙でいい」と拒否される
- 施設長から「家族から指摘があったから停止」と言われる

#### Fact / Assumption / Unknown / Risk のサマリー

```
- Fact: 5
- Assumption: 8
- Unknown: 2
- Risk: 2
```

**注意**: Assumption が Fact を上回っている。**このペルソナは仮説段階**。Validation Plan で確かめる優先順位を立てる必要がある。

#### このペルソナで未検証の最大の論点
1. **[Risk]** 「紙の自由度を犠牲にすると速さを失う」が本当か（紙との入力時間比較が最重要）
2. **[Assumption]** 「夜勤者の既読が分かること」が本当に主要価値か
3. **[Unknown]** 共用端末でのログイン運用が現実的に回るか

## 2. Validation Plan

### 検証対象: ペルソナ 1（Segment 1）

#### 検証する状況
夜勤への引き継ぎを書く 17〜19時、現場のPCまたは共用タブレットから。実際の利用者ケース（マスキング済み）を使う。

#### 検証する仮説
「構造化されたフォームでの引き継ぎ入力が、(a) 紙ノートと同等以下の時間で完了し、(b) 後から検索できることが書き手にとって有意義に感じられ、(c) 夜勤者の既読が分かることが価値として響く」

#### 仮説のうち最も Risk が高い部分
**[Risk]** (a) 紙と同等以下の時間に収まること — フォームの構造化と速さは原理的にトレードオフ。ここが外れると Segment 1 の主要 JTBD（速く書ける）が満たせない。

#### 反証条件（これが起きたら仮説は外れた）
- 5人中3人以上が「紙のほうが楽」と回答
- 平均入力時間が紙の **1.5倍以上**
- 「項目が多すぎる」というネガティブ反応が3人以上
- 「既読が分かっても自分のモチベーションは変わらない」と4人以上が回答

#### 確証条件（これが起きたら仮説は支持された）
- 5人中4人以上が「これなら使ってもよい」と回答
- 平均入力時間が紙と同等以下
- 「あとから検索できる」が肯定的に評価される（4人以上が言及）
- 翌週も自発的に使い続ける（5人中3人以上）

#### 検証手段
- **プロトの形式**: Figma クリッカブル → 反応次第で軽量実機
- **検証セッション**: 1on1 30分、実際の引き継ぎシナリオ（マスキング済み）で入力体験
- **サンプル数**: Segment 1 から **5人**（Nielsen 経験則）
- **計測**: 入力時間（ストップウォッチ）、満足度（5段階）、自由記述

#### この検証で答えられない問い（次フェーズに残す）
- 3ヶ月後も使い続けるか（長期検証は別途）
- 施設長の導入判断に効くか（Stage 1 論点2に依存）
- 夜勤スタッフ側の受け取り体験（受け手側UIは Segment 3 = OOS 領域）

## 3. Out-of-scope Rationale

Stage 1 で Secondary / Out of scope にしたセグメントについて、**なぜ今回外したか** と **どんな条件が変わったら再考すべきか** を書く。

### Segment 2: 書く習慣がない現場リーダー（C 型）

**今回外した理由**: 1人サンプル（C）のみで、ペルソナの解像度が低い。「書く動機」が前提として存在するか未検証で、ここを Primary にすると検証対象が「書くツール」ではなく「書く文化」になり、プロト1サイクルでは答えが出ない。

**再考トリガー**:
- 並走する簡易インタビュー（3人×15分）で「実は記録したい欲求があった」が複数人から出てきた
- Segment 1 のプロトが想定より早く合格基準を満たし、リソースに余裕ができた
- 競合・市場の動きで Segment 2 への先行投資が戦略的に必要になった

### Segment 3: 夜勤スタッフ（受け手）

**今回外した理由**: 今回のプロト目的が「書く側のUI」だから。受け手側UIに着手すると、書き手の課題への注意が分散する。

**再考トリガー**:
- Segment 1 検証で「書いたが見られない」が問題の本質だと判明
- 夜勤スタッフ側に補助インタビューを入れて、受け取り体験が予想以上に深刻と分かった場合

### Segment 4: 施設長（決裁者）

**今回外した理由**: 直接UIを作らない。価値訴求のコミュニケーションフェーズで扱う対象。

**再考トリガー**:
- Primary 検証が成功し、導入フェーズに進む段階
- 施設長への簡易ヒアリングで「現場ツールに加えてレポート機能が必須」と判明

## このサンプルの読み方（Stage 2 観点）

- **Stage 1 を前提としている** — 単独で読まず、`output-example-stage1.md` とセットで見る
- **Persona Deep Dive で「インタビュー引用 + 推測の境界」を明示** — Fact（直接発言）と Assumption（書き手の延長）を分けて書く
- **Validation Plan に数字** — 反証条件・確証条件が具体的に書かれており、検証が「確認作業」にならない
- **Out-of-scope に再考トリガー** — 外した理由とともに、いつ見直すかも書かれている（凍結しない）
