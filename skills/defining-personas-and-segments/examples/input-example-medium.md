# 社内データカタログツール 検討メモ

## 背景

社内のデータ活用が広がるにつれて、各チームが BigQuery 上のテーブルを参照するようになった。
ただ「どこに何のデータがあるか」がブラックボックスで、データチームへの「これってどのテーブル?」という問い合わせが増えている。
社内向けデータカタログツールの導入を検討中。

## ヒアリング (3名)

### A: マーケアナリスト (入社2年目)

- 必要なテーブルを探すときは、データチームの Slack チャンネルに DM を投げるか、知っている人に直接聞く
- Confluence のドキュメントは「半分くらい間違っている」ので信用していない
- 「テーブル名で検索できて、最終更新日とオーナーが分かればすごく助かる」
- 新人の頃は誰に聞けばいいかも分からず、Slack で悩んでいた

### B: データエンジニア (ベテラン)

- 自分用には不要(テーブル構造は頭に入っている)
- 「他チームから "これ何?" って聞かれる回数を減らしたい」が動機
- 「カタログがあれば自己解決してもらえる」ことを期待
- ただし「メンテナンスを自分たちがやるなら、そのコストが見合うか分からない」

### C: 経営企画 (月数回データに触る)

- 自分で SQL は書かない
- ダッシュボードを見るが「欲しい数字がそこに無い」ことが多く、結局 A に依頼する
- 「テーブル名を見ても、自分には使えない」
- 「こういう数字が欲しい」で検索して「このダッシュボード」と出る方が嬉しい

## 定量メモ (ざっくり)

- 本番系テーブル数: 約400
- 中間テーブル: 約1200
- データチームへの問い合わせ: 月50件程度
- ダッシュボード数: 把握できていない(おそらく200くらい)

## チーム内で迷っていること

- データを「書く人」(アナリスト/エンジニア)向けか、「読みたい人」(経営企画など)向けか
- 検索体験の問題か、コミュニケーション(誰に聞けばいい?)の問題か
- カタログのメンテナンスコストは誰が払うのか
- 既存の Confluence ドキュメントは廃止するのか、併用するのか
- OSS の DataHub / Amundsen を入れるか、自前で軽く作るか

## まだ確認できていないこと

- B のチームへの問い合わせ件数の内訳(質問の種類)
- 過去にカタログ的なものを誰かが試して失敗していないか
- 経営企画チームの他メンバーが C と同じニーズを持つかは未確認
- セキュリティ・権限管理の要件
