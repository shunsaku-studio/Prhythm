# product-vision-and-concept

## 概要

プロダクトの**ビジョン・コンセプト**を、対話で一緒に言語化するスキル。AI が代筆するのではなく、質問で素材を引き出し、複数案を提示し、磨き込んで**ユーザー自身が「これだ」と言える言葉**に着地させる。

- **入力:** プロダクト名（あれば）、概要、言語（日本語・英語）
- **出力:** **一行ステートメント** + **Why / Who / What / How it's different** の最終稿、品質チェック結果
- **スコープ外:** 競合調査、ロードマップ・OKR 策定、GraphQL / ドメイン設計、Moore 型エレベーターピッチ単体

## 利用メリット

- **自分たちの言葉で説明できる** — 代わりに書かれた文案ではなく、対話を通じて納得できる表現に着地する
- **チームの北極星が一本になる** — 一行ステートメントと詳細コンセプトが揃い、判断の基準を共有しやすい
- **白紙から書き始めやすい** — 一度に届く質問と方向性の選択肢で、考えが動き出す
- **複数案で「違う」がはっきりする** — 2〜3 パッケージ + 統合案を見比べることで、本当に欲しいトーンや粒度が見える
- **PRD 冒頭がそのまま使える** — 企画書の導入に転用できる粒度まで整えられる
- **既存文案の弱点が具体的に分かる** — 貼り付けたビジョンを第三者目線で診断し、改善の方向が示される

## 利用シーン

- **プロダクトの説明がうまくできないとき** — 一言で言えず、機能説明に逃げがちな状態を整理したい
- **チーム内で方向性の認識がバラバラなとき** — 壁に貼れる一行と、PRD 用コンセプトを揃えたい
- **新規企画の PRD 冒頭を書く前に、存在理由を固めたいとき** — Why / Who / What / 差別化まで文章化したい
- **書いたビジョンを第三者目線で見てほしいとき** — 「これどう思う？」「改善したい」と既存文案を渡したい
- **競合調査やスキーマ設計に入る前に、プロダクトの Why を先に決めたいとき** — 後工程に渡せる言語化が欲しい

依頼の例: 「Feedit のビジョンを一緒に作って」「この一文、どう思う？ 改善したい」

## 使い方

**新規作成** — プロダクト名や概要を伝える。詳しく説明済みならそのまま質問フェーズへ進む:

```
ビジョンステートメントを作りたい
社内で感謝を伝えるアプリを作っている。一行と PRD 冒頭が欲しい
PRD の冒頭を書きたい
```

**既存文案のレビュー** — ビジョンやコンセプトを貼り付けてフィードバックを依頼:

```
このビジョン、どう思う？
[文案を貼る] — 改善したい
```

**磨り込み** — ドラフト提示後、トーンや長さの調整を依頼:

```
統合案が近いけど、もうちょっとカジュアルにしたい
この一行で、Concept の Who から練り直して
```

**保存** — 完成後、Markdown ファイルへのまとめを依頼:

```
ファイルにまとめて
```

## 構成

```
product-vision-and-concept/
├── README.md
├── SKILL.md
└── references/
    ├── frameworks.md    # 一行ステートメントと Concept の書き方
    ├── examples.md      # 実在企業・良い/悪い事例
    └── antipatterns.md  # 失敗パターンと改善例（品質チェック用）
```

## 前提条件

- 特別な MCP・スクリプトは不要 — 対話と Markdown 出力のみ
- プロダクト名・ターゲット・課題など、分かっている範囲を伝えられると早い（未確定でも `(仮)` で進行可能）
- スキル配置: Prhythm の `skills/` または `~/.cursor/skills/`

## 注意事項

- **競合・参考サービスの調査**が主目的のときは使わない — `competitive-research` を使う
- **GraphQL / ドメインスキーマ設計**が主目的のときは使わない — `ooui-graphql-modeling` を使う
- **ロードマップ・OKR・数値目標**だけを決めたいときは対象外 — 言語化（ビジョン・コンセプト）がゴールのときに使う
- 回答が薄くても進める — 未確定部分は `(仮)` と明示し、後から差し替え可能
- セッション終了時は**一行 + Concept の最終稿**と品質チェックを必ず出す
