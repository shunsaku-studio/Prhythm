# ジャーニーマップ生成ガイド

## 概要

このskillはジャーニーマップ（JM）の生成を支援する。2つのモードを持ち、サブコマンドとして使い分ける。

### モード一覧

| モード | コマンド例 | インプット | アウトプット |
|-------|-----------|-----------|-------------|
| **mode:asis** | 「As-Is JMを作って」 | ヒアリングメモ、ペルソナ情報 | As-Is JM → インサイト → HMW |
| **mode:tobe** | 「To-Be JMを作って」 | As-Is JMのアウトプット＋HMW優先度 | To-Be JM → 対比サマリー → コアシーン候補 |

### モード判定ルール

- ユーザーの指示に「As-Is」「現状」「ヒアリングから」などが含まれる → **mode:asis**
- ユーザーの指示に「To-Be」「理想」「HMWをもとに」などが含まれる → **mode:tobe**
- モード指定がない場合 → **mode:asis** をデフォルトとする
- mode:tobe実行時にAs-Is JM・HMW優先度が渡されていない場合 → 実行せず、mode:asisの実行を促す

### なぜ2モードに分かれているか

As-Is→HMWとTo-Beの間には、チームの意思決定（どのHMWを優先するか）が必要になる。この判断をスキップしてTo-Beに進むと、現状の課題を起点としない「ウィッシュリスト的なTo-Be」が生まれるリスクがある。mode:asisは必ずHMW提示で停止し、チームの合意を待つ。

## JMの定義と役割

### ジャーニーマップとは

ユーザーの行動・感情・思考・状況を時系列で可視化するツール。核心は「As-Is（現状）とTo-Be（理想）の差分から、どこに介入すべきか（課題・機会）を特定すること」にある。

### よくある誤用（注意）

- ❌ 「プロトタイプで体験させたいコアシーンを特定するため」にJMを使う → コアシーンの特定はJMから導出される後段の作業であり、JM自体の目的ではない
- ❌ As-IsとTo-Beを同時に作る → As-Isの課題が見えてはじめて意味のあるTo-Beが描ける
- ❌ 感情を数値スコア（1〜5）で表現する → 定性的なJMでは数値より言葉で表現した方がリアルになる

## 共通ルール

以下のルールはmode:asis・mode:tobeの両方に適用する。

### 台本形式の記述ガイドライン

- 演劇の台本のように、そのペルソナのシチュエーションをありありと描く
- 状況描写は五感に訴えるレベルで書く（場所、時間帯、周囲の様子）
- 発話は「本当にこの人が言いそうな言葉」にする
- 感情は数値スコアではなく言葉で表現する
- ヒアリング情報に含まれるエピソード・発言は最大限活用する
- 推測で補った箇所は `※推測` と明示する

### フェーズの設計ルール

- フェーズ数は固定しない。そのペルソナの体験を描くのに必要な数だけ設ける
- 粒度は大きめでよい（「認知」「初回利用」「継続」など）
- ペルソナの行動や感情をありありと思い描けることが最優先

### 各フェーズの共通構造

```
【フェーズ名】

状況：
  その人がいる場所・時間帯・周囲の環境を具体的に描写する。
  映像として頭に浮かぶレベルの解像度で書く。

発話 / 内心の声：
  その人が実際に口にする言葉、または頭の中のつぶやきを一人称で書く。

行動：
  その場面でとる具体的なアクションを、順序がわかるように書く。

感情：
  感情の変化を言葉で表現する。数値スコアは使わない。

タッチポイント：
  この場面で接触するサービス・画面・人・モノを列挙する。
```

### アウトプット形式

- メインアウトプット：台本形式のMarkdownテキスト
- オプション（ユーザーの要望に応じて出力）：
  - フェーズ一覧のMarkdownテーブル（Notion・Confluenceなどに貼る用）
  - CSV形式（FigJam・Miroへのインポート用）

## mode:asis — As-Is JM生成

### インプット

以下のいずれか、または複数の組み合わせ。形式は問わない。

- ヒアリングメモ（箇条書き、自由記述など）
- インタビュー議事録・会話ログ
- ペルソナ情報（名前、属性、役割、背景など）
- 既存の調査結果・アンケートデータ

### 情報不足時の対応

**必須情報（これがないとJMを作れない → 生成せず質問する）**
- ペルソナの基本属性（誰か、どんな立場か）
- 現状の体験の概要（何をしていて、何に困っているか）

**あると望ましい情報（なければ推測で補い `※推測` と明示する）**
- 具体的な発話・エピソード
- 利用しているツール・サービス・タッチポイント
- 体験の時間軸（日常的か、月次か、特定イベント時か）
- 周囲の環境・登場人物（上司、同僚、顧客など）

### 生成ステップ

#### Step 1: ペルソナとシナリオの確認

インプットからペルソナ情報を整理し、以下を確認する。

```
【ペルソナ確認】
- 名前（仮名可）：
- 属性（年齢、職種、役割など）：
- シナリオの範囲（どの体験を描くか）：
```

不明点があればこの段階で質問する。

#### Step 2: As-Is JMの生成（台本形式）

共通ルールの台本形式に従い、各フェーズに以下を追加する。

```
課題：
  この場面で生じている摩擦・不満・諦め・非効率を具体的に書く。
```

#### Step 3: インサイト抽出

As-Is JMの全フェーズを俯瞰し、表面的な課題の奥にある構造的な問題を3〜5個のインサイトとして抽出する。

**インサイトの基準**
- ヒアリングの生の声をそのまま転記するのではなく、「なぜそうなっているか」の一段深い解釈にする
- 複数のフェーズにまたがるパターンを見つける
- 「〜が〜と分断されている」「〜する機会がない」「〜がブラックボックスになっている」のような構造的な記述にする

**形式**

```
【インサイト】
1. （構造的な問題の記述）
2. （構造的な問題の記述）
3. （構造的な問題の記述）
```

#### Step 4: HMW（How Might We）問いへの変換

各インサイトを「どうすれば〜できるか？」という問いに変換する。

**HMWの基準**
- 広すぎず狭すぎない粒度にする（解決策を限定しないが、方向性は示す）
- 1つのインサイトから1〜2個のHMWを導出する
- 「どうすれば」で始める

**形式**

```
【HMW】
1. どうすれば、〜できるか？
2. どうすれば、〜できるか？
3. どうすれば、〜できるか？
```

#### ⛔ Step 4完了後：必ず停止する

HMWの提示後、Claudeは以下のメッセージを出力し、To-Beの生成には進まない。

```
---
✅ As-Is JM・インサイト・HMWの生成が完了しました。

【次のステップ】
チームでHMWの優先度を合意してください。
- どのHMWを最優先で取り組むか？
- To-Beで描くべき方向性はどれか？

優先度が決まったら、このskillの mode:tobe でTo-Be JMを生成できます。
---
```

### mode:asis チェックリスト

- [ ] ペルソナの状況が映像として浮かぶレベルで描かれているか
- [ ] 各フェーズの発話が「本当にこの人が言いそうな言葉」になっているか
- [ ] As-Isの痛点が具体的なシーンとして描かれているか（抽象的な課題リストになっていないか）
- [ ] インサイトが表面的な課題の転記ではなく、構造的な解釈になっているか
- [ ] HMWが広すぎず狭すぎない粒度か
- [ ] 推測で補完した箇所にすべて `※推測` が付いているか
- [ ] To-Beに進まず、停止メッセージで終わっているか

## mode:tobe — To-Be JM生成

### インプット

以下のすべてが必要。

- **As-Is JM**（mode:asisのアウトプット）
- **インサイト一覧**（同上）
- **HMW一覧と優先度**（チームが合意したもの。最優先のHMWが明示されていること）

### 情報不足時の対応

**必須情報（これがないとTo-Beを作れない）**
- As-Is JMの全文（フェーズ・状況・課題が含まれていること）
- チームが選んだ最優先のHMW（最低1つ）

HMWの優先度がチームで合意されていない場合、To-Beの生成には進まず、先にチーム内での議論を促す。

### 生成ステップ

#### Step 1: インプットの確認

```
【To-Be生成の前提】
- ペルソナ：（As-Is JMから引き継ぎ）
- 最優先HMW：（チームが選んだもの）
- 副次HMW：（余裕があれば取り込むもの）
- As-Isの主要課題：（To-Beで解決すべきポイント）
```

#### Step 2: To-Be JMの生成（台本形式）

共通ルールの台本形式に従い、各フェーズに以下を追加する。

```
As-Isからの変化：
  このフェーズでAs-Isから何が変わったかを明示する。
  どのインサイト・HMWに対応しているかも記述する。
```

**To-Be固有のガイドライン**
- As-Isと同じフェーズ構造をベースにする（Before/Afterの対比を明確にするため）
- ただし、課題が解決されることでフェーズが統合・消滅・新規追加されることは自然
- 「魔法のように全部解決される」夢物語にしない。HMWで選んだ範囲の課題が解決された現実的な理想像にする
- 新しいタッチポイントが登場する場合は、ユーザーがそれをどう体験するかを具体的に描く
- HMWで扱わない課題は、To-Beでも残っていてよい

#### Step 3: As-Is / To-Be 対比サマリー

```markdown
| フェーズ | As-Is（現状） | To-Be（理想） | 変化のポイント |
|---------|-------------|-------------|--------------|
| （フェーズ名） | （一文で要約） | （一文で要約） | （何が変わったか） |
```

#### Step 4: コアシーン候補の提示

To-Be JMのフェーズの中から、プロトタイプで優先的に検証すべきシーンを2〜4個提示する。

**選定基準**
- As-Isとの差分が最も大きいシーン（変化のインパクトが大きい）
- ユーザーの感情が最もポジティブに変わるシーン（体験価値が高い）
- 技術的・ビジネス的に検証が必要なシーン（不確実性が高い）

**形式**

```
【コアシーン候補】

候補1: （フェーズ名 — シーンの一言説明）
  選定理由：（上記の基準のどれに該当するか）
  検証したいこと：（プロトで何を確かめたいか）

候補2: （フェーズ名 — シーンの一言説明）
  選定理由：
  検証したいこと：
```

#### Step 4完了後：次のステップを提示する

```
---
✅ To-Be JM・コアシーン候補の生成が完了しました。

【次のステップ】
1. チームでコアシーン候補を確認し、プロトタイプで検証するシーンを選定してください
2. 選定したコアシーンをもとに、プロトタイプの設計に進んでください

コアシーン選定のポイント：
- 1〜2シーンに絞る（検証の焦点を明確にするため）
- 「これが上手くいくか不安」なシーンを優先する（確信があるシーンはプロトで検証する価値が低い）
---
```

### mode:tobe チェックリスト

- [ ] To-BeがHMWを起点にしているか（ウィッシュリストになっていないか）
- [ ] As-Isとの対比が各フェーズで明確になっているか
- [ ] ペルソナの状況が映像として浮かぶレベルで描かれているか
- [ ] 発話が「本当にこの人が言いそうな言葉」になっているか
- [ ] HMWで扱わない課題が無理に解決されていないか
- [ ] 対比サマリーで変化のポイントが端的にまとまっているか
- [ ] コアシーン候補に選定理由と検証したいことが書かれているか
- [ ] 推測で補完した箇所にすべて `※推測` が付いているか
