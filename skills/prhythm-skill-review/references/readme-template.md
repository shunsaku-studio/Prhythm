## 利用メリット

**問い:** このスキルを使うと、何が嬉しい？ 作業や判断がどう楽になる？

- **成果・状態の変化**を書く（「〜できる」「〜が揃う」「〜がブレにくい」）
- **ユーザー・チームが得る価値**を先に。仕組みの説明は後（`SKILL.md` / `docs/` に任せる）

**書かないもの:**

- トリガー条件・スラッシュコマンド・フェーズ名
- 命名規則、検証スクリプト、ゲート手順などエージェント向け実装詳細
- 読者に意味が伝わらない内部用語

**例:**

```
❌ Phase ゲートで承認するので手戻りが減る
❌ validate スクリプトで SDL を機械チェックできる

✅ 画面のたたき台になる — オブジェクトと操作が揃うと、何を作るか見える
✅ 実装前に形を固められる — 後からドメインがブレにくい
```

- **{得られる価値 1}** — {一言で補足}
- **{得られる価値 2}** — {一言で補足}
- **{得られる価値 3}** — {一言で補足}

## 利用シーン

**問い:** どんな状況のとき、このスキルを使う？ **利用メリット**と対応させる。

- **状況・困りごと**を先に書く（「〜なとき」「〜したい」）
- 各項目は **メリットの価値**に結びつく一文で締める
- 可能ならメリットと **1:1 で対応**させる

**書かないもの:**

- 技術操作の列挙（型追加、再検証、フェーズ実行など）→ **使い方**へ
- スラッシュコマンド・トリガー語の一覧 → **使い方**へ
- メリットと無関係な汎用フレーズ

**例:**

```
❌ schema.graphql の type・関係を追加・修正するとき
❌ トリガー例: domain schema, GraphQL SDL

✅ アイデアはあるが、何が「もの」で何が「操作」か曖昧なとき — 言葉にして共有したい
✅ 画面を描き始める前に、全体像を揃えたいとき — たたき台が欲しい
```

末尾に **依頼の例** を1行足してよい:

```
依頼の例: 「ブログアプリのスキーマを一緒に設計して」
```

- **{状況 1}** — {メリットに結びつく一文}
- **{状況 2}** — {メリットに結びつく一文}
- **{状況 3}** — {メリットに結びつく一文}
